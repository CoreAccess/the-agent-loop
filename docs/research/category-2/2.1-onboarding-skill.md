# Research: 2.1 — Human-in-the-Loop Onboarding Skill

Date: 2026-04-27
Status: Re-grill complete. Decisions recorded below.

---

## What This Sub-Category Covers

The conversational skill that initiates a project. A human and agent work through a structured set of questions together to distill the project down before a single line of code is written. The output feeds directly into 2.2 (Shared Vision & Constitution).

This is the most novel part of the framework — no existing tool does this well. grill-me (our current skill) is 7 lines. We need something far more structured.

---

## Key Findings

### 1. Initial metadata conditions all downstream responses

HITL academic research found that a "static personalized pipeline" — conditioning every subsequent response on onboarding metadata — achieved the highest adaptability score of any approach tested. The key insight: the quality of the onboarding conversation directly determines the quality of everything that follows.

What metadata matters for a coding project:
- Project type (new project vs. existing project)
- Tech stack and existing architecture (if existing project)
- Team size and AI tool(s) being used
- Launch constraints and hard deadlines
- Product rules and non-negotiables
- Known unknowns (what the user is uncertain about)
- What "done" looks like for this project

### 2. Cognitive friction is valuable — don't shortcut it

DK Consulting research (2026): "Critical thinking and generative AI require human-in-the-loop cognitive friction." The onboarding skill should not make it too easy for the user to skip hard questions. Ambiguity left in the onboarding conversation becomes ambiguity baked into the project.

Pattern: ask the hard question, give a concrete recommendation (like grill-me does), then let the user push back. Force a decision before moving on.

### 3. BMAD's Analyst role is the closest existing model

BMAD assigns an Analyst agent specifically for requirements gathering and research before planning begins. The Analyst answers five critical questions before anything is designed:
1. What is the system context?
2. Are requirements clear and unambiguous?
3. What are concrete examples of success?
4. What are the acceptance criteria?
5. What are the edge cases?

Our onboarding skill should adapt this for project initialization — not feature requirements, but project requirements.

### 4. Harness engineering: human checkpoints between phases

Red Hat (2026): "Harness engineering" inserts explicit human review checkpoints between AI-driven phases. The onboarding skill should have at least two checkpoints:
1. After the questions are answered — human confirms the distillation before the shared vision doc is drafted
2. After the shared vision doc is drafted — human confirms before bootstrap generation begins

### 5. grill-me is a valid structural pattern but needs depth

Our existing grill-me skill (7 lines) works because:
- One question at a time prevents overwhelm
- Providing a recommendation with each question reduces cognitive load
- Walking decision branches systematically ensures completeness

What it's missing for project initialization:
- Domain-specific question sets (web app vs. data pipeline vs. API vs. CLI tool)
- Structured output format (the vision doc template to populate)
- Explicit resolution of dependencies between answers
- New vs. existing project mode switch
- A clear ending state (when is the conversation "done"?)

## Patterns to Borrow

| Pattern | Source | What to Take |
|---------|--------|-------------|
| One question at a time + recommended answer | grill-me | Keep exactly as-is |
| Onboarding metadata conditions all downstream responses | HITL research | Structure the questions so their answers become variables in the bootstrap |
| Five critical questions (context, clarity, examples, criteria, edge cases) | BMAD Analyst | Adapt for project scope, not feature scope |
| Human checkpoint before artifacts are generated | Red Hat harness engineering | Two explicit confirmation points |
| Cognitive friction is intentional | HITL research | Don't let users skip or rush |
| New vs. existing project mode | BMAD, D³ framework | Branch the question set based on this first answer |

## Re-Grill Decisions

- **Question flow:** Fixed core of 5-6 universal questions + adaptive branches by project type. Universal core: project type, new vs. existing project, tech stack, launch definition, hard constraints.
- **"I don't know" handling:** TBDs become Pending Decisions tagged with context clues so far. AI researches as conversation develops. End of onboarding: AI closes remaining TBDs through research + human confirmation. Every question pre-classified as Blocking (can't start without it) or Non-blocking (sensible default applies). Non-blocking TBDs always have a visible fallback default. Blocking TBDs remaining after research = explicit gate before bootstrap runs.
- **Project depth:** Skill auto-detects the right depth from onboarding answers — no explicit tier selection exposed to the user. If uncertain, falls back to one clarifying question: "Quick Start" or "Full Setup." No tier vocabulary for the user to learn.

## Questions Remaining for Deep Research

- What is the minimum viable question set that captures enough context without being exhausting? (research: what questions do experienced devs ask before starting a project?)
- How do we handle onboarding for an existing project — the skill needs to inspect the existing codebase, not just ask questions
- What's the right format for the output? (feeds into 2.2)
