## Key Findings

### 1. Three distinct documents serve three distinct roles

Research surfaced a clear separation that we should adopt:

| Document | Role | Who edits | Nature |
|----------|------|-----------|--------|
| **Project Constitution / Shared Vision** | What the project is and governing principles | Human only | Stable, rarely changes |
| **AGENTS.md** | How the agent behaves in this project | Human (with AI drafting) | Evolves with project conventions |
| **CONTRACT.md** | Complexity ceilings by area — the "naughty list" | Human only (AI can only advise) | Enforced, actively amended |
| **Planning docs (STATUS, DECISIONS, BACKLOG)** | Ongoing state, decisions, queue | Human + AI | Changes frequently |

The author of CONTRACT.md (discussdontcode.com) makes this memorable: "AGENTS.md is a wishlist. Read once, soon forgotten. CONTRACT.md is a hitlist."

**Recommendation:** Our bootstrap should generate all four. The onboarding skill populates the Constitution and seeds CONTRACT.md. AGENTS.md is generated from the Constitution + a base template. Planning docs are stubs.

### 2. CONTRACT.md is the missing piece in most frameworks

The "complexity ceiling" pattern is underrepresented in the ecosystem. Most frameworks (spec-kit, BMAD) focus on what to build — not on what NOT to build, or how complex it's allowed to get.

CONTRACT.md works by:
- Listing major project areas ("rocks") with explicit complexity ceilings
- Using present-tense constraints: "All storage in one relational DB. No Redis. For NOW."
- "For NOW" language acknowledges constraints are temporary but deliberate
- Only humans can amend it — AI can only flag when it's being approached
- Functions as a Van Halen "brown M&M" check: if the AI knows about this constraint and follows it, it read the whole contract

This should be a first-class artifact in our framework.

### 3. spec-kit's constitution is a good but narrow model

spec-kit's `constitution.md` establishes:
- Code quality standards
- Testing standards
- UX/performance principles
- Tooling conventions

What spec-kit's constitution does NOT cover (and ours should):
- Product scope boundaries (what the project is and isn't)
- Known constraints and non-negotiables
- Complexity ceilings (CONTRACT.md territory)
- New vs. existing project context
- Team and stakeholder context
- Launch definition (what does "done" mean?)

### 4. Agentic constitutions in enterprise contexts

CIO.com (2026): enterprise agentic constitutions include:
- Sandbox of trust: threshold-based triggers where agents do heavy lifting but require human approval before final execution
- Reasoning traces: agents explain why they're taking actions before taking them
- Authentication against the constitution before interacting with core infrastructure
- Audit trail for compliance

These are enterprise patterns but the core ideas translate to single-developer projects:
- The agent should be able to cite the constitution when making decisions
- The constitution creates accountability ("I did X because the constitution says Y")

### 5. The shared vision doc and the constitution are two separate things

After researching, these should be distinct artifacts:

**Shared Vision Doc** (ephemeral, produced during onboarding):
- Conversational summary of what was decided in the onboarding skill
- Written in plain language the human recognizes as their own intent
- Confirmed by human before any files are generated
- Not preserved long-term — it graduates into the Constitution

**Project Constitution** (durable, generated from the vision doc):
- Structured, versioned, machine-readable
- Lives at `/constitution.md` in project root
- Sections: Project Identity, Product Scope, Governing Principles, Stack Contract, Launch Definition, Complexity Ceilings
- Referenced by AGENTS.md and planning docs

## Proposed Constitution Structure

Based on research synthesis:

```markdown
# Project Constitution

## 1. Project Identity
What this project is in one paragraph. Core purpose, core user.

## 2. Product Scope
What's in and what's explicitly out. Non-negotiables.

## 3. Governing Principles
Code quality, testing, UX, performance, security standards.

## 4. Stack Rationale
Why these technology choices were made — the constraints and reasoning behind the stack.
(The specific tech stack facts — e.g. "React, Tailwind, TypeScript" — live in AGENTS.md. The constitution holds the why, so agents can consult it for context when needed.)

## 5. Launch Definition
What "done" means for v1. Success criteria. 

## 6. Complexity Ceilings
Area-by-area complexity limits. "For NOW" constraints.
(Amend deliberately. Only humans edit this section.)

## 7. Authority and Escalation
When the agent must stop and ask. What requires human approval.
```

## Re-Grill Decisions

- **Constitution vs AGENTS.md split:** Constitution = why (rationale, reasoning, governing principles). AGENTS.md = what (facts, rules, tech stack specifics). No duplication by design. Agent reads AGENTS.md first; consults constitution only when it needs the reasoning behind a decision. Constitution wins on any conflict.
- **Complexity ceilings (CONTRACT.md):** Deferred — flagged for deep research. Initial position: full agent autonomy inside the codebase; only stop and ask for very large reworks or significant refactors. Research needed to sharpen what "large rework" means in practice.
- **Amendment rule:** AI may propose constitution or CONTRACT.md amendments with rationale. AI may never self-amend. Human approves → AI makes the edit → recorded in DECISIONS.md.
