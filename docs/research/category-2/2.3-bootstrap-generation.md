# Research: 2.3 — Bootstrap Generation

Date: 2026-04-27
Status: Re-grill complete. Decisions recorded below.

## What This Sub-Category Covers

What gets generated after the constitution is confirmed. The complete set of files, directories, and scaffolding that sets a project up for AI-assisted development. This runs once — it's the transition from "blank repo" to "framework-equipped project."

---

## Key Findings

### 1. Bootstrap must handle new and existing projects differently

Research is clear that starting a new project and adding to an existing one are fundamentally different modes requiring different approaches. BMAD is the only framework studied that explicitly handles both.

**New project (starting from scratch):**
- Sequential artifact-driven pipeline
- Generate all files from constitution answers
- No codebase to inspect first
- Order: constitution → AGENTS.md → planning docs → directory structure → README

**Existing project (adding to an existing codebase):**
- Must inspect the codebase BEFORE generating anything
- Red Hat harness engineering: use LSP/MCP to build a "Repository Impact Map" first
- BMAD's "Flattener" tool: onboards AI into established systems by mapping existing patterns
- D³ Framework (arxiv): Discover phase first — understand existing architecture before defining anything
- Risk: generating AGENTS.md without inspecting the codebase produces rules that conflict with existing conventions

**Our onboarding skill (2.1) must ask new vs. existing project as its first question.** The rest of the bootstrap branches from that answer.

### 2. Bootstrap depth is auto-detected, not user-selected

The onboarding skill infers the right depth from what it learns during the conversation — no explicit tier selection is shown to the user. If the skill is uncertain, it asks one clarifying question: "Quick Start" or "Full Setup?" No tier vocabulary for the user to learn.

Internally, three depth levels shape what gets generated:

| Internal depth | Use case | What it generates |
|----------------|----------|------------------|
| **Quick** | Bug fixes, small features on existing project | constitution.md stub, CONTRACT.md stub, task log |
| **Standard** | New product or MVP | Full bootstrap: constitution, AGENTS.md, planning docs, memory, directory structure |
| **Full** | Complex system with multiple domains | Standard + architecture docs, ADRs, domain-specific AGENTS.md per directory, skills installation |

### 3. What a complete Standard bootstrap should generate

Based on synthesis across spec-kit, BMAD, gstack, and the former local FilmGap agent contract reference:

```
project-root/
├── constitution.md           # Project identity, scope, principles, ceilings
├── CONTRACT.md               # Complexity ceilings (naughty list)
├── AGENTS.md                 # Agent behavior contract (generated from constitution)
├── memory/
│   ├── MEMORY.md             # Memory index
│   └── project.md            # Initial project state memory
├── docs/
│   ├── STATUS.md             # Current priorities and known gaps
│   ├── DECISIONS.md          # Major scope decisions log
│   ├── BACKLOG.md            # Post-launch ideas
│   └── architecture/         # Architecture docs (stubbed)
└── .claude/                  # Or .gemini/ or agent-specific dir
    └── skills/               # Installed skills
```

For brownfield, also generate:
```
docs/
└── existing-architecture.md  # Map of current codebase produced during inspection
```

### 4. Each generated file should be pre-populated, not empty

Empty templates are ignored. The bootstrap should:
- Use the constitution answers to populate AGENTS.md with project-specific rules
- Pre-populate STATUS.md with initial known gaps identified during onboarding
- Seed DECISIONS.md with the major decisions made during the vision session
- Write MEMORY.md with the initial project memory entry

The grill-me session we just completed is the proof: we generated real content, not stubs.

### 5. Persistent artifacts over chat history

BMAD's core principle: "Documents versioned like code, not chat history that disappears." Every bootstrap artifact should be:
- In the repo (not in AI memory or chat history)
- Markdown (human-readable, diffable)
- Versioned via git
- Referenced explicitly by AGENTS.md

### 6. Skills should be installed as part of bootstrap

gstack, gemini-skills, and mattpocock all use CLI-based skill installation (`npx skills add`, etc.). Our bootstrap should install the framework's core skills into the agent's skills directory as part of the bootstrap step — not as a separate manual step.

Skills to install at bootstrap:
- `onboard` (the init skill itself, for re-running if needed)
- `plan` (task planning skill)
- `review` (code review skill)
- `reflect` (session retrospective skill)
- Any project-type-specific skills (web, API, CLI, data pipeline)

### 7. Quality gates between bootstrap phases

Red Hat harness engineering: human review checkpoint between phases prevents errors from propagating. Bootstrap should have two explicit gates:
1. **Gate 1:** Human reviews constitution before any files are generated
2. **Gate 2:** Human reviews the generated file list before skills are installed

## Patterns to Borrow

| Pattern | Source | What to Take |
|---------|--------|-------------|
| New vs. existing project branching | BMAD, D³, Red Hat | First question in 2.1 determines bootstrap mode |
| Auto-detected depth (internal Quick/Standard/Full) | BMAD | Skill infers depth — no tier vocabulary exposed to user |
| Repository Impact Map before generation | Red Hat harness engineering | For existing projects: inspect codebase with LSP/MCP first |
| Pre-populated artifacts, not empty templates | gstack, BMAD | Constitution answers fill in the templates |
| Persistent docs versioned like code | BMAD | All artifacts go in repo, not just memory |
| CLI-based skill installation | gemini-skills, mattpocock | Bootstrap installs skills automatically |
| Human approval gates | Red Hat | Two explicit checkpoints during bootstrap |

## Re-Grill Decisions

- **Existing file handling:** Blocking files (constitution.md, AGENTS.md, CONTRACT.md) — show a diff and ask the user: overwrite, skip, or merge. Everything else — skip if it already exists and note what wasn't touched. Bootstrap never silently overwrites.
- **Mid-phase interruption:** Agent writes two things at session end: a STATUS.md handoff note (where we are, what's done, what's next) and a phase checklist (items ticked as complete). Next session reads both and resumes from the last open checklist item.

## Questions Remaining for Deep Research

- Exact file-by-file content for each generated artifact (what does a generated AGENTS.md look like for a Go web app vs. a Python data pipeline?)
- What does the existing-project codebase inspection step look like in practice for each agent (Claude Code, Gemini, Codex)?
- Should bootstrap be re-runnable — can you run it again on an existing project without destroying what's there?
- Skill installation: how do we handle cross-agent portability (Claude skills ≠ Gemini skills)?
